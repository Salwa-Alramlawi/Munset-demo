Track 04: Agent-to-Agent (A2A)

1 ماذا يفعل النظام؟ — What the Agents Do

مُنصِت (Munset) منظومة وكلاء ذكية تعمل بالكامل عبر Cloud APIs (بدون أي موديل محلي أو GPU) لأتمتة توثيق وتحليل الجلسات القضائية في المحاكم السعودية.

يرفع المستخدم (قاضي أو كاتب ضبط) تسجيلاً صوتياً للجلسة، فينتج النظام:

  • نص مُفرّغ مع تحديد المتحدثين (قاضي، مدعي، مدعى عليه، محامين)
  • تحليل قانوني — ادعاءات، دفوع، أدلة، مواد نظامية، تناقضات
  • محضر جلسة رسمي مُهيكل + ملخص تنفيذي + إجراءات قادمة
  • تقرير جودة — نسبة اكتمال، ملاحظات، تحقق من المواد النظامية
  • مساعد تفاعلي (شات بوت) — القاضي يسأل أي شي عن الجلسة ويحصل إجابة فورية
  • كشف تناقضات عبر الجلسات — مقارنة أقوال نفس الطرف في جلسات مختلفة
القيمة: توفير 70-80% من وقت التوثيق، دقة أعلى، تحليل قانوني فوري، ودعم التحول الرقمي في القطاع العدلي السعودي (رؤية 2030).

2 قرارات كل وكيل — What Decisions Each Agent Makes

كل وكيل مستقل — يستقبل مهمة عبر بروتوكول A2A، يتخذ قرارات متخصصة، ويرسل النتائج:

الوكيل القرارات المستقلة تفاعلات A2A التقنية
🎙️ التفريغ الصوتي
  • تحديد هوية كل متحدث عبر الـ timestamps
  • دمج نتائج ASR مع diarization (midpoint matching)
  • ربط speaker IDs بالأدوار (قاضي، مدعي، إلخ)
 يرسل النص المفرّغ لوكيل التحليل Groq Whisper API + pyannote.ai API
⚖️ التحليل القانوني
  • تصنيف نوع القضية من المحتوى
  • التمييز بين الادعاءات والدفوع
  • تحديد المواد النظامية والتحقق من انطباقها
  • كشف التناقضات داخل الجلسة
  • بناء تسلسل زمني للأحداث
 يرسل التحليل لوكيل التلخيص
يستقبل طلبات توضيح من وكيل المراجعة ويرد عليها 		Claude API
📝 التلخيص
  • هيكلة المحضر الرسمي بصيغة قانونية
  • تحديد ما يدخل الملخص التنفيذي وما يبقى في التفاصيل
  • استخراج الإجراءات القادمة والنقاط المعلّقة
 يرسل المحضر لوكيل المراجعة
يستقبل ملاحظات جودة من وكيل المراجعة 		Claude API
✅ مراجعة الجودة
  • تقييم نسبة الاكتمال (0-100%) بمقارنة المحضر بالنص الأصلي
  • تحديد درجة خطورة كل ملاحظة (عالية / متوسطة / منخفضة)
  • قرار التوجيه: أي وكيل سابق يُرسل له الملاحظة
    (missing_info → التلخيص، contradiction → التحليل القانوني)
  • طلب توضيحات من الوكلاء السابقين عند الحاجة
 يرسل feedback للتلخيص
يرسل clarification requests للتحليل القانوني 		Claude API
💬 المساعد التفاعلي
  • فهم سؤال القاضي في سياق بيانات الجلسة
  • اختيار مصدر الإجابة (النص، التحليل، أو المحضر)
  • الاحتفاظ بسياق المحادثة للأسئلة المتتابعة
  • رفض الإجابة إذا المعلومة غير موجودة في البيانات
 يستقبل أسئلة المستخدم عبر A2A Claude API
القرار الأهم — Autonomous Routing: وكيل المراجعة لا يكتفي بالفحص — بل يقرر بشكل مستقل أي وكيل سابق يتواصل معه وبأي نوع رسالة (feedback vs. clarification). هذا يجعل النظام Agent-to-Agent حقيقي، وليس مجرد pipeline أحادي.

3 نوع النظام — Multi-Agent with Bidirectional A2A

مُنصِت نظام Multi-Agent مكوّن من 5 وكلاء متخصصين يتواصلون عبر بروتوكول Agent-to-Agent (A2A) مخصص ثنائي الاتجاه. كل شي يعمل عبر Cloud APIs بدون أي موديل محلي.

ليش Multi-Agent وليس Single-Agent؟

  • فصل التخصصات: كل وكيل عنده prompt متخصص ومهمة محددة — وكيل واحد يحاول يسوي كل شي بينتج نتائج أضعف.
  • تواصل ثنائي الاتجاه: الوكلاء يراجعون ويحسّنون شغل بعض — وكيل المراجعة يرجع بملاحظات للوكلاء السابقين.
  • قابلية التطوير: كل وكيل يُحدَّث مستقلاً (مثلاً: تبديل Groq Whisper بنموذج أفضل بدون تغيير بقية النظام).
  • شفافية كاملة: سجل A2A يوفر audit trail — أي قرار اتخذه أي وكيل موثّق ومتتبّع. مهم جداً في نظام قضائي.

الوكلاء الخمسة

🎙️ Transcription Agent CLOUD API

APIs: Groq Whisper API (ASR) + pyannote.ai API (speaker diarization)

Output: نص مُفرّغ مع timestamps وهوية كل متحدث

⚖️ Legal Analysis Agent CLOUD API

API: Claude API مع prompt متخصص في النظام القضائي السعودي

Output: JSON — نوع القضية، ادعاءات، دفوع، أدلة، مواد نظامية، تناقضات، تسلسل زمني

📝 Summary Agent CLOUD API

API: Claude API مع persona كاتب ضبط

Output: محضر جلسة رسمي + ملخص تنفيذي + قائمة إجراءات قادمة

✅ QA Agent CLOUD API

API: Claude API + A2A bidirectional (feedback & clarification requests)

Output: نسبة اكتمال، قائمة ملاحظات، تحقق من المواد النظامية، ملاحظات للوكلاء السابقين

💬 Chatbot Agent CLOUD API

API: Claude API مع كامل بيانات الجلسة + ذاكرة محادثة

Output: إجابات فورية على أسئلة القاضي مع الاحتفاظ بسياق الأسئلة المتتابعة

ليش A2A وليس مجرد Pipeline؟

أغلب أنظمة الوكلاء تمرر بيانات باتجاه واحد (1 ← 2 ← 3 ← 4). بروتوكول A2A في مُنصِت يدعم تواصل ثنائي الاتجاه:

  • تدفق أمامي (data): التفريغ ← التحليل ← التلخيص ← المراجعة ← الشات بوت
  • تدفق عكسي (feedback): وكيل المراجعة ← وكيل التلخيص (ملاحظات جودة)
  • تدفق عكسي (clarification): وكيل المراجعة ← وكيل التحليل (طلبات توضيح)
  • تتبع الرسائل: كل رسالة لها msg_id و parent_id — سلسلة محادثة كاملة قابلة للتتبع
  • أولويات: طلبات التوضيح تُعلَّم كـ high priority
  • إحصائيات: عدد الرسائل لكل وكيل (مُرسلة، مُستقبلة، توضيحات، ملاحظات) معروضة في الواجهة
